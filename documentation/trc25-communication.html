Communication Dofbot - Convoyeur

Le Dofbot est positionné en fin de convoyage afin d’assurer le tri des déchets transportés sur le convoyeur. Lorsqu’un déchet atteint la zone finale du convoyeur, le système de convoyage doit notifier le Dofbot de sa présence afin de permettre une prise en charge rapide et précise de l’objet.
La performance globale du système repose en grande partie sur la qualité, la fiabilité et la rapidité de la communication entre ces deux sous-systèmes, qui doivent fonctionner de manière synchronisée pour garantir un tri efficace.

Afin de garantir une interaction efficace et adaptée aux différents scénarios de fonctionnement, trois types de communication ont été étudiés et testés :

  • communication I2C
  • communication sans fil
  • communication via une broche

1. Communication I2C

La communication I²C (Inter-Integrated Circuit) est un protocole de communication série synchrone à courte distance, reposant sur une architecture maître–esclave. Dans ce système, le Dofbot joue le rôle de maître, tandis que le convoyeur agit comme esclave.

    Le bus I²C repose sur deux lignes principales:
  • SDA(Serial Data) : ligne de données
  • SCL(Serial Clock) : ligne d’horloge
Avantages Inconvenients
  • Câblage réduit(trois fils dont deux de communication)
  • Bonne fiabilité pour des distances courtes
  • Communication bidirectionnelle
  • Distance de communication limitée
  • Sensibilité au bruit électrique
  • Dépendance au maître

Les connexions entre le Dofbot et le convoyeur sont réalisées comme suit :

Dofbot(Maître) Convoyeur(esclave)
SDA SDA
SCL SCL
GND GND

NB:Il est indispensable d’avoir une masse commune (GND) entre les deux systèmes afin d’assurer une référence électrique identique et garantir le bon fonctionnement de la communication.

a. Adaptation de niveau logique

Le Dofbot fonctionne avec un niveau logique de 3,3V, tandis que le microcontrôleur du convoyeur (Arduino) utilise des niveaux logiques de 5V. Cette différence de tension peut entraîner des erreurs de communication, voire endommager les composants.
Pour assurer une communication fiable et sécurisée, un convertisseur de niveau logique bidirectionnel est utilisé entre les lignes SDA et SCL(Voir section Convoyeur 2.0). Ce dispositif permet :

  • d’adapter les niveaux de tension entre 3,3 V et 5 V,
  • de protéger les entrées/sorties des deux systèmes,
  • de garantir l’intégrité des signaux I²C.

b. Programmation

Afin d’établir la communication I²C, il est nécessaire de connaître l’adresse I²C de l’esclave. Cette adresse permet au maître, en l’occurrence le Dofbot, d’identifier correctement le microcontrôleur du convoyeur (Arduino) sur le bus I²C.
La détection de cette adresse peut être effectuée côté Dofbot à l’aide d’un scan du bus I²C, en exécutant la commande suivante : 


i2cdetect -y -r -a 1

Ayant identifié l’adresse I²C de l’esclave, le Dofbot peut alors établir la communication et envoyer des commandes ou recevoir des informations en utilisant le protocole I²C. Le code suivant illustre l’initialisation de la communication I²C, l’envoi d’une commande simple depuis le Dofbot vers le microcontrôleur du convoyeur, ainsi que la réception de la réponse envoyée par l’esclave.

i2c_dofbot.py 

import smbus   #bibliotheque i2c
import time

# Bus I2C du Jetson
bus = smbus.SMBus(1)
ARDUINO_ADDR = 0x15


def send_command_and_read():
    try:
        #Envoyer une commande
        bus.write_byte(ARDUINO_ADDR, 5)
        print("Commande envoyée: 5")

        #Lire la réponse de l'esclave (1 octet)
        time.sleep(1)  # petit délai pour que l'Arduino prépare la réponse
        resp = bus.read_byte(ARDUINO_ADDR)
        print("Réponse reçue de l'Arduino:", resp)
    except Exception as e:
        print("Erreur I2C:", e)


while True:
    send_command_and_read()
    time.sleep(1)  #boucle toutes les secondes

volatile byte lastCmd = 0;

  Serial.begin(9600);

  Wire.onRequest(requestEvent);

void loop() {

}

void receiveEvent(int howMany) {
  while (Wire.available()) {
    lastCmd = Wire.read();
  }
  Serial.print("Donnée recue : ");
  Serial.println(lastCmd);
}

void requestEvent() {
  int data = 2;          // donnée envoyée

  Wire.write(data);         // envoi I2C
  reqCount++;

  Serial.print("Requete n° : ");
  Serial.println(reqCount);
  Serial.print("Envoyé au maître : ");
  Serial.println(data);
  Serial.println("------------------------------------");
}

Les résultats expérimentaux montrent que la transmission des données du Dofbot vers le convoyeur est fonctionnelle. Toutefois, la réception des données émises par l’esclave vers le maître n’aboutit pas, la valeur retournée étant systématiquement nulle.

Plusieurs tests ont été réalisés dans le but de déterminer la cause du dysfonctionnement observé. La communication entre le convoyeur et un autre microcontrôleur Arduino a notamment été validée, confirmant le bon fonctionnement de l’esclave I²C. Les connexions matérielles ont été revérifiées, un délai de réponse a été introduit dans le code, et les broches de communication ont été modifiées afin d’écarter toute erreur liée au câblage ou à la configuration matérielle.
Malgré ces ajustements, le problème persiste. Il apparaît donc que le dysfonctionnement soit lié à la réception des données au niveau du Dofbot.

2. Communication sans fil

Afin de contourner les limitations rencontrées avec la communication I²C, une communication sans fil a été mise en place en parallèle, entre le Dofbot et le convoyeur, à l’aide du module ESP-01 (ESP8266). Cette approche permet d’assurer un échange de données bidirectionnel, sans liaison filaire, tout en offrant une plus grande flexibilité d’intégration .

Le module ESP-01 est basé sur le microcontrôleur ESP8266, intégrant une interface Wi-Fi et supportant les protocoles TCP/IP. Il permet ainsi l’envoi et la réception de données via un réseau local sans fil.

    L’architecture retenue repose sur une communication client–serveur :
  • Dofbot : serveur Wi-Fi
  • Convoyeur : client Wi-Fi

Le convoyeur notifie le Dofbot lorsqu’un déchet est détecté en fin de convoyage. Le Dofbot analyse cette information et déclenche l’action de tri appropriée.

Avantages Inconvenients
  • Suppression des contraintes de câblage
  • Communication bidirectionnelle fiable
  • Plus grande portée
  • Communication bidirectionnelle fiable
  • Dépendance à la qualité du réseau Wi-Fi
  • Complexité logicielle plus élevée

NB: Le convoyeur(ESP-01 relié en UART avec l'arduino) et le dofbot doivent être connectés au même réseau Wi-Fi afin de permettre l’échange de données.

a. Adaptation de niveau logique

Tout comme le dofbot, l'ESP-01 fonctionne avec un niveau logique de 3,3V. Un convertisseur de niveau logique est donc nécessaire.(Voir section Convoyeur 2.0).

b. Programmation

La programmation du système sans fil se divise en trois parties distinctes, correspondant aux différents composants impliqués :

  • ESP-01: relié à l’Arduino via UART, il gère la communication avec le Dofbot en tant que client Wi-Fi.
  • Arduino: interface avec les capteurs et actionneurs, envoi des données du convoyeur via l’ESP-01.
  • Dofbot: agit comme point d’accès Wi-Fi (AP) et reçoit les notifications du convoyeur pour le tri des déchets.

#include <SoftwareSerial.h>

// RX, TX
SoftwareSerial espSerial(4, 7);

#define pinStart 2

#define LED_R 9
#define LED_G 10
#define LED_B 11

unsigned long lastInterrupt = 0;
volatile bool is_on = false;

// variables pour gestion du timing
unsigned long previousMillis = 0;
int colorState = 0;
const unsigned long interval = 2000; // intervalle de 2 secondes

void setup()
{
    Serial.begin(9600);        // PC
    espSerial.begin(9600);     // ESP

    pinMode(LED_R, OUTPUT);
    pinMode(LED_G, OUTPUT);
    pinMode(LED_B, OUTPUT);
    setColor(0, 0, 0); // éteint la LED

    pinMode(pinStart, INPUT);
    attachInterrupt(digitalPinToInterrupt(pinStart), interrupt_routine, CHANGE);

    delay(500);
    is_on = digitalRead(pinStart);
}

void loop() {
    unsigned long currentMillis = millis();

    if (is_on) {
        if (currentMillis - previousMillis >= interval) {
            previousMillis = currentMillis;

            // changer la couleur en fonction de colorState
            switch(colorState) {
                case 0:
                    setColor(250, 0, 0); // rouge
		            espSerial.write("rouge");
                    Serial.println("Sent: rouge");
                    break;
                case 1:
                    setColor(0, 250, 0); // vert
		            espSerial.write("vert");
                    Serial.println("Sent: vert");
                    break;
                case 2:
                    setColor(0, 0, 250); // bleu
		                espSerial.write("bleu");
                    Serial.println("Sent: bleu");
                    break;
            }

            colorState++;
            if (colorState > 2) colorState = 0;
        }
    } else {
        setColor(0, 0, 0); // LED éteinte
        previousMillis = currentMillis; // réinitialiser le timing
        colorState = 0;
    }
}

void setColor(unsigned int red, unsigned int green, unsigned int blue)

}

3. Communication UART

La communication UART (Universal Asynchronous Receiver Transmitter) repose sur une liaison série asynchrone point à point entre deux systèmes. Elle permet l’échange de données sans horloge partagée, la synchronisation étant assurée par des paramètres de communication communs.
Dans ce projet, la communication UART est établie directement entre le convoyeur et le Jetson Nano, assurant une transmission simple, rapide et fiable des informations nécessaires au fonctionnement du système de tri.

Avantages Inconvenients
  • Mise en œuvre simple
  • Communication stable et fiable
  • Faible latence
  • Portée limitée par la longueur du câble
  • Communication point à point uniquement
  • Dépendance à la connexion physique

a. Connexion matérielle

La liaison UART est réalisée à l’aide d’un câble USB–série, reliant le microcontrôleur du convoyeur au Jetson. Une fois connecté, le périphérique série est reconnu sous Linux sous la forme : 


/dev/ttyUSB0

Cette interface est utilisée par le programme exécuté sur le Jetson pour recevoir les données émises par le convoyeur.

NB: L’utilisation d’un câble USB–série garantit une adaptation automatique des niveaux logiques et simplifie l’intégration matérielle.

b. Programmation

    Deux programmes ont été développés afin de tester la communication UART filaire entre le convoyeur et le Jetson Nano (Dofbot) :
  • uart_arduino.ino : exécuté sur l’Arduino du convoyeur, il permet l’envoi périodique d’un message via l’interface UART.
  • uart_dofbot.py : exécuté sur le Jetson Nano, il assure la réception des données transmises par le convoyeur via le port série (/dev/ttyUSB0).

4. Conclusion générale

Dans le cadre de ce projet, trois types de communication entre le convoyeur et le Dofbot ont été étudiés, implémentés et testés : la communication I²C, la communication sans fil via l’ESP-01 (Wi-Fi) et la communication UART filaire.

a. Communication I2C

La communication I²C, bien que théoriquement adaptée pour des échanges rapides à courte distance, n’a pas permis d’assurer une communication bidirectionnelle fonctionnelle dans le contexte de ce projet. Malgré plusieurs tests et vérifications (câblage, délais, configuration logicielle), la réception des données au niveau du Dofbot est restée non fonctionnelle. Cette communication n'est donc pas retenue.

b. Communication sans fil

La communication sans fil via l’ESP-01 s’est révélée fonctionnelle et a permis un échange bidirectionnel de données entre le convoyeur et le Dofbot. Toutefois, cette solution introduit une latence liée au réseau Wi-Fi ainsi qu’une complexité logicielle plus élevée, ce qui peut constituer un inconvénient pour des actions nécessitant une réponse immédiate.
Malgré ces contraintes, un code de communication TCP a été développé et testé côté Dofbot afin de valider le fonctionnement global de cette approche. Ce programme met en œuvre un serveur TCP, capable de recevoir les notifications envoyées par le convoyeur (via l’ESP-01), de détecter la présence d’un objet et de gérer un timeout en cas d’absence de nouvelles données.

Le code suivant illustre l’implémentation du serveur TCP exécuté sur le Dofbot :

tcp_server.py 

#!/usr/bin/env python
# coding: utf-8

import socket
import time


class tcp_Server:
    def __init__(self, host='0.0.0.0', port=5001, timeout=2.5):
        self.host = host
        self.port = port
        self.timeout = timeout

        self.server = None
        self.conn = None
        self.addr = None

        self.presence = False
        self.last_msg_time = 0

    def start_server(self):
        self.server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        self.server.bind((self.host, self.port))
        self.server.listen(1)

        print(f"Serveur TCP prêt sur {self.host}:{self.port}")
        self.conn, self.addr = self.server.accept()
        print("Connexion Arduino :", self.addr)

        self.conn.setblocking(False)

    def listen(self):
        try:
            data = self.conn.recv(1024)
            if data:
                cmd = data.decode().strip()
                if cmd == "piece":
                    self.presence = True
                    self.last_msg_time = time.time()
        except BlockingIOError:
            pass

        if self.presence:
            if time.time() - self.last_msg_time > self.timeout:
                self.presence = False

    def close(self):
        if self.conn:
            self.conn.close()
        if self.server:
            self.server.close()

c. Communication UART

La communication UART filaire entre le convoyeur et le Jetson Nano s’est révélée simple, fiable et rapide, répondant efficacement aux besoins de transmission des informations critiques du système. Contrairement à la communication sans fil, cette solution ne dépend pas d’un réseau et ne présente qu’une latence négligeable, ce qui la rend particulièrement adaptée aux actions nécessitant une réponse immédiate.
Dans ce cadre, un code de communication UART a été développé côté Jetson Nano afin de valider le bon fonctionnement de cette liaison.

uart_Arduino.py 

#!/usr/bin/env python
# coding: utf-8

import serial
import time

class uart_Arduino:
    def __init__(self, serial_port="/dev/ttyUSB0", baudrate=9600, timeout=2.5):
        self.serial_port = serial_port
        self.baudrate = baudrate
        self.timeout = timeout

        self.arduino = None
        self.presence = False
        self.last_msg_time = 0

    def start(self):
        try:
            self.arduino = serial.Serial(self.serial_port, self.baudrate, timeout=1)
            time.sleep(2)  # temps pour l'initialisation de l'Arduino
            print(f"Connexion UART établie sur {self.serial_port} à {self.baudrate} bauds")
        except Exception as e:
            print(f"Erreur de connexion UART : {e}")

    def listen(self):
        if self.arduino and self.arduino.in_waiting > 0:
            try:
                data = self.arduino.readline().decode().strip()
                if data == "piece":
                    self.presence = True
                    self.last_msg_time = time.time()
            except Exception as e:
                print(f"Erreur lecture UART : {e}")

        # Timeout pour remettre presence à False
        if self.presence and (time.time() - self.last_msg_time > self.timeout):
            self.presence = False

    def close(self):
        if self.arduino:
            self.arduino.close()
            print("UART fermée")

d. Conclusion

En conclusion, la communication UART filaire a été retenue comme solution principale pour ce projet, tandis que la communication sans fil constitue une alternative flexible et évolutive. La communication I²C, quant à elle, n’a pas été retenue en raison de son manque de fiabilité dans cette configuration.
